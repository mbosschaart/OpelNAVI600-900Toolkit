# Opel Navi 600/900 Firmware Toolkit

A reverse engineering toolkit for the **Opel/Vauxhall Navi 600 and Navi 900**
head units (GM-GE platform, manufactured by Bosch). Provides tools to
decompress, inspect, disassemble, modify, and repackage the firmware — turning
opaque binary blobs into something you can understand and work with.

The head unit runs a **MIPS-IV 32-bit Little-Endian** processor and boots from
NAND flash. The firmware consists of a bootloader (`dragon.bin`) and 13
application modules, each stored as LZO-compressed ELF binaries inside a
proprietary container format called XOZL.

This toolkit cracks that format open.

## The Toolkit

### xozl_tool.py — XOZL Container Tool

The central tool. The `.out` firmware files use a proprietary container called
**XOZL** that wraps LZO1X-compressed ELF binaries. This tool handles the full
lifecycle: inspect headers, decompress to raw ELF, and recompress modified ELFs
back into valid XOZL containers.

```bash
# Inspect: see what's inside a .out file
python3 tools/xozl_tool.py info ProcHMI.out

# Extract: decompress to raw ELF (with CRC32 verification)
python3 tools/xozl_tool.py extract ProcHMI.out ProcHMI.elf

# Pack: compress a (modified) ELF back to XOZL
python3 tools/xozl_tool.py pack ProcHMI_patched.elf ProcHMI_patched.out --ref ProcHMI.out
```

The `--ref` flag is key for packing: it copies all header constants and the
version trailer from the original `.out`, only updating the three fields that
change (compressed size, decompressed size, CRC32). The result is structurally
identical to a factory file. Packing includes automatic round-trip verification.

Full format specification and implementation details:
[`tools/docs/xozl_tool_README.md`](tools/docs/xozl_tool_README.md)

### mips_disasm.py — Interactive MIPS Disassembler

Point-and-shoot disassembler for examining specific address ranges in any MIPS
binary. Uses instruction-at-a-time decoding so it never stops on data embedded
in code sections.

```bash
# Look at a specific function
python3 tools/mips_disasm.py ProcHMI.elf --offset 0x4f0700 --size 0x100

# With raw hex encoding alongside
python3 tools/mips_disasm.py ProcHMI.elf --offset 0x9a87a0 --size 0x70 --hex

# Bootloader analysis (MIPS64 mode)
python3 tools/mips_disasm.py dragon.bin --offset 0x4ab50 --size 0x80 --mips64

# Attempt string reference resolution
python3 tools/mips_disasm.py ProcHMI.elf --offset 0x4f0500 --size 0x200 --strings
```

Works on decompressed ELFs (from `xozl_tool.py extract`) and raw binaries
(`dragon.bin`). MIPS32 LE by default, `--mips64` for the bootloader.

### disasm_pack.py — Batch Firmware Disassembler

Processes an entire firmware update tree: decompresses every `.out` module,
parses ELF headers, disassembles the `.text` section, and outputs structured
results. Also handles `dragon.bin`.

```bash
# Disassemble everything from a firmware directory
python3 tools/disasm_pack.py --adit-dir /path/to/g__eeu10

# Single module only
python3 tools/disasm_pack.py --adit-dir /path/to/g__eeu10 --module ProcHMI.out

# Full executable segment instead of just .text
python3 tools/disasm_pack.py --adit-dir /path/to/g__eeu10 --full

# Skip the bootloader
python3 tools/disasm_pack.py --adit-dir /path/to/g__eeu10 --skip-dragon
```

Output per module: `elf_meta.json` (ELF headers, all sections and segments) +
`text.asm` (complete `.text` disassembly). The `.asm` files are plain text,
one instruction per line with address prefix — perfect for `grep`/`rg`:

```bash
# Find all calls to a specific function
rg "jal.*0x4f5c88" 208_source/modules/ProcHMI/text.asm

# Find string-loading patterns
rg "lui.*0x9a" 208_source/modules/ProcHMI/text.asm
```

Full documentation: [`tools/docs/disasm_README.md`](tools/docs/disasm_README.md)

### uli_tool.py — ULI Resource Archive Tool

Inspects and extracts ULI resource archives (`.uli` files), which contain
fonts, images, and other UI assets used by the head unit.

Documentation: [`tools/docs/uli_tool_README.md`](tools/docs/uli_tool_README.md)

### validate_xozl.py — XOZL Integrity Validator

Comprehensive validation suite for verifying XOZL files are correct and
bootloader-compatible.

```bash
python3 tools/validate_xozl.py ProcHMI_patched.out \
    --elf ProcHMI_patched.elf \
    --ref original/ProcHMI.out
```

Tests: header structure, LZO decompression, CRC32, ELF magic, size match,
byte-for-byte cross-validation, header comparison, and original-compatibility
check (confirms the original also uses standard LZO1X).

## Firmware Architecture

### Boot Process

```
NAND Flash
  │
  ▼
dragon.bin (bootloader, MIPS64)
  ├── Hardware init: DRAM, PLL/clocks, DMA controller
  ├── FlashFX NAND filesystem with bad block management
  ├── Dual-partition failsafe: origin vs changed, error counters
  ├── For each .out module:
  │     ├── Read from NAND
  │     ├── Check first 4 bytes:
  │     │     "XOZL"  → LZO1X decompress → load as ELF
  │     │     "\x7fELF" → load directly as raw ELF
  │     │     "ULI "  → resource archive handler
  │     └── Load ELF segments into RAM
  └── Transfer control to application modules
```

### Module Map

The firmware is split across 13 modules plus the bootloader. They communicate
via a **CCA (Component Communication Architecture)** message bus built on top
of a **CSFW (Component Software Framework)**.

| Module | Size | Role |
|--------|------|------|
| **dragon.bin** | 1.9 MB | Bootloader — NAND flash, LZO decompression, ELF loader, USB update |
| **sysprogosalio** | 1.7 MB | OS abstraction — threads, mutexes, GPIO, RTC, file I/O, crypto |
| **sysprog1** | 2.2 MB | Audio/display HW drivers — DAC, I2S, DMA, SPI, ASRC, fonts |
| **sysprog2** | 1.5 MB | Storage middleware — NAND, FAT, ATA/IDE, DMA, block devices |
| **sysprogcal** | 454 KB | Audio calibration — stream routing, signal path configuration |
| **sysprogperf** | 395 KB | Performance monitor — task profiling, latency, watchdog |
| **ProcBase** | 5.6 MB | Base framework — CCA/CSFW messaging, security, amplifier, VDL |
| **ProcHMI** | 15.6 MB | User interface — iPod/USB, Bluetooth, nav UI, phone, menus |
| **ProcNav** | 16.5 MB | Navigation engine — routing, map DB, TMC traffic, GPS, guidance |
| **ProcSDS** | 6.3 MB | Speech dialog — voice recognition, TTS, grammars, ASR engine |
| **ProcMW** | 4.8 MB | Middleware — UDS diagnostics, ACR, system diagnostic handlers |
| **ProcMM** | 4.0 MB | Multimedia — media browser, audio source management |
| **ProcMP1** | 1.6 MB | Media player — USB/CD/DVD playback engine, metadata DB |
| **ProcMap** | 2.4 MB | Map renderer — tile engine, MDM, POI display, viewport control |

Detailed analysis of each module — classes, functions, subsystems, and key
string references — is in [`208_source/MODULES.md`](208_source/MODULES.md).

## XOZL Format

The `.out` firmware files use a 36-byte header followed by a standard LZO1X
compressed ELF binary and a version trailer:

```
Offset  Size  Field
──────  ────  ─────────────────────────────────────
0x00    4     Magic: "XOZL" (LE 0x4c5a4f58)
0x04    4     Reserved (0)
0x08    4     Version major (1)
0x0C    4     Version minor (2)
0x10    4     Reserved (0)
0x14    4     Header length (0x24 = 36)
0x18    4     Decompressed ELF size (bytes)
0x1C    4     Compressed payload size (bytes)
0x20    4     CRC32 of decompressed content (init=0)
0x24    N     LZO1X compressed payload
0x24+N  var   Version trailer (e.g. "GM10.8V208")
```

The compression is **standard LZO1X** — confirmed by successfully decompressing
all original factory firmware with `python-lzo`. The CRC32 covers the
**decompressed** content, not the compressed stream.

### Bootloader File Dispatch

The bootloader at `0x4ab68` in `dragon.bin` identifies file types by magic:

```
0x4ab68:  lui  $v0, 0x4c5a      ─┐
0x4ab6c:  ori  $v0, $v0, 0x4f58  │ $v0 = "XOZL"
0x4ab70:  beq  $v1, $v0, 0x4ac40 │ → XOZL handler (decompress + load)
                                  │
0x4ab78:  ori  $v0, $v0, 0x4c55  │ $v0 = "ULI "
0x4ab7c:  beql $v1, $v0, 0x4ac44 │ → ULI handler (resource archive)
                                  │
0x4ab84:  lui  $v0, 0x464c       │
0x4ab88:  ori  $v0, $v0, 0x457f  │ $v0 = "\x7fELF"
0x4ab8c:  beq  $v1, $v0, 0x4adb0 │ → raw ELF handler (direct load)
```

## Example: iPod/iPhone Connectivity Patch

As a practical demonstration of the toolkit, this repository includes a
complete binary patch that fixes a widespread iPhone connectivity issue
on the Navi 600/900.

### The Problem

iPhones connected via USB frequently fail with "This accessory is not
supported." The MFi authentication handshake fails intermittently due to timing,
and the firmware has no retry logic — it logs the error and gives up.

### The Fix

A 93-byte MIPS code patch injected into `ProcHMI.elf` that intercepts both
auth failure handlers and adds automatic disconnect → delay → reconnect retry
logic (up to 5 attempts).

### Building the Patch

```bash
pip install python-lzo
bash tools/build_patch.sh /path/to/original/ProcHMI.out
```

Runs: extract → patch → verify → repack → validate. Output: `build/ProcHMI_patched.out`.

### Patch Details

| Location | Description |
|----------|-------------|
| `0x004f0714` | Auth CP Error handler → jump to retry code cave |
| `0x004f077c` | Auth Failed handler → jump to retry code cave |
| `0x009a87a0` | Code cave: 26 MIPS instructions implementing retry logic |

The code cave performs: load retry counter → disconnect iAP session →
busy-wait ~150ms → clear init flag → reconnect → return. After 5 failures,
falls through to original behavior. Full technical breakdown in the patch
tool's source: [`tools/patch_ipod_auth_retry.py`](tools/patch_ipod_auth_retry.py).

**Compatibility:** Verified against firmware v2.08 (`GM10.8V208`). The patch
tool checks expected bytes at each site before writing, so it will refuse to
patch an incompatible version.
